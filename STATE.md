## Что закрыто в этапе 7 (итог)

Полная цепочка живая: `jobs.run_poll_cycle` раз в минуту тянет курсор
из БД → детектор → рассылка → курсор обратно; `jobs.run_due_date_cycle`
раз в сутки напоминает о дедлайнах. Оба висят на APScheduler, зависимости
собирает lifespan.

| Модуль | Роль |
|---|---|
| `status_resolver.py` | TTL-кэш «id статуса → имя» |
| `poller.py` | `PollCursor` + `poll_recent_changes` — чистый детектор |
| `sent_notifications.py` | `is_already_sent` / `mark_sent` — идемпотентность |
| `dispatcher.py` | событие → routing → рендер → MAX → отметка |
| `polling_state.py` | `load_cursor` / `save_cursor` — мост ORM ↔ `PollCursor` |
| `jobs.py` | `JobDeps` + оба цикла |
| `user_mapping.py` | «Redmine-юзер → MAX-юзер» для @упоминаний |

Принятые решения (чтобы не переобсуждать):

- **`monotonic`, не `datetime.now()`** — настенные часы прыгают (NTP,
  перевод времени) и ломают арифметику TTL.
- **Ошибка Redmine при протухшем кэше → исключение наружу**, stale-значение
  НЕ отдаём.
- **Промах по свежему кэшу → сразу `None`, без рефреша.** Иначе событие
  с удалённым статусом било бы по API на каждом проходе.
- **Окно `updated_on` выбирает задачи, новизну решают id.** Окно нужно,
  потому что эндпоинта `/journals.json` в Redmine нет. Но `issue.id` и
  `journal.id` глобально монотонны — детекция по ним не зависит от часов.
- **Холодный старт = `last_check_at is None`**, а не пустые `last_seen_*`.
  Первый цикл ставит baseline и не шлёт ничего. Иначе тихий Redmine
  (пустое окно → максимумов нет) залипал бы в «первом запуске» вечно.
- **Удалённый статус → событие пропускается с `log.warning`**, а не
  заглушка «#8» и не падение цикла.
- **`mark_sent` ПОСЛЕ отправки** (at-least-once): дубль в чате раздражает,
  потерянное уведомление о проде вредит. Диспетчер коммитит по событию,
  а не батчем — отправка в MAX необратима.
- **Курсор сохраняется ПОСЛЕ рассылки.** Упал цикл в середине — курсор
  старый, события придут снова, их отсечёт идемпотентность.
- **Дедуп дедлайнов включает `notified_due_date`** — одно напоминание на
  каждое значение срока, сдвиг срока даёт новое напоминание.
- **Один чат на проект** — решено 15.07.2026. Схема `SentNotification`
  без `chat_id` этого достаточно. Появится второй чат — вернуться к
  вопросу: сейчас отметка ставится на событие целиком, и если из двух
  чатов упал один, повтора не будет ни для одного.
- **Журналы тянутся карточкой задачи, а не списком** (7h). Redmine
  поддерживает `include=journals` только для `/issues/{id}.json`;
  в списке молча игнорирует. Отсюда N+1, обойти нельзя.
- **Текст из Redmine экранируется** фильтром `| md`. Тема и комментарии
  пишут люди, а шаблон заворачивает их в `*...*`.
- **Время рендерится в `Settings.timezone`** фильтром `| dt`. Голых
  `strftime` по datetime в шаблонах быть не должно — напечатают UTC.
- **Эмодзи статусов/приоритетов — фильтрами**, а не `{% if %}` в
  шаблонах: один словарь на четыре шаблона, иначе разъедутся.
- **Приватная заметка прячет и вложения.** Имя файла выдаёт содержание
  не хуже текста.
- **Ссылка «Открыть в Redmine» убрана** (решение Leo, «пока не нужно»).
  `redmine_base_url` и `REDMINE_BASE_URL_PUBLIC` оставлены — может
  вернуться.
- **Таблицы в комментариях приезжают кашей** — MAX их не рендерит.
  Решено оставить как есть: если начнут пользоваться часто, вернёмся
  (вариант — обернуть в код-блок, если MAX умеет ```).

тестов. Все — потому что моки писал я сам и подтверждал ими свои же

3. **Прикреплённый файл молча пропадал** — событие требовало непустой
   `notes`, а Redmine пишет вложение в `details`.
4. **Поля в `new_issue` слипались** — `trim_blocks` съедал переносы
   после inline `{% if %}`.
5. **Markdown из темы ломал вёрстку** — `*обрыв*` закрывал жирный.

Плюс инфраструктурное: `.env` отстал от `.env.example` (`MAX_BOT_TOKEN`
вместо `MAX_TOKEN`, не было Redmine-переменных), а тесты конфига читали
личный `.env` разработчика.

**Мораль на будущее:** пока код не сходил в живой API, зелёные тесты
означают только «мои догадки самосогласованы».

## Активные грабли

Свежие наблюдения. Когда этап закроется целиком и «повторяющаяся»
грабля себя не проявит ещё пару этапов — переносим её в `CLAUDE.md`
или просто удаляем.

- **Консольного `sqlite3` в Windows нет.** Команды диагностики в
  шпаргалках — через `uv run python -c "import sqlite3; ..."`. Написал
  памятку с `sqlite3 file.db "запрос"` — она бы врала с первой строки.
  На Linux работает, но команды должны быть одинаковыми в обеих средах.

- **MAX не отдаёт «всех пользователей».** Только участников чатов, где
  состоит бот (`GET /chats/{id}/members`). Историю сообщений боту тоже
  не показывает: `GET /messages` вернул 0 при `messages_count: 18`.

- **Тест, написанный по коду, охраняет баг.** `test_new_issue` ожидал
  `"14.07.2026 15:30"` — то есть ровно UTC из Redmine. Когда починили
  рендер времени, тест честно покраснел, **защищая ошибку**. Он писался
  по факту («что вывелось, то и правильно»), а не по требованию
  («человек видит своё время»). Пиши тест от требования, иначе он
  зафиксирует любой баг, который был на момент написания.

- **Redmine отдаёт время в UTC** (`"created_on": "...Z"`), а `due_date` —
  календарная дата без времени. Первое рендерим через `| dt`
  (таймзона из конфига), второе — как есть.

- **`include=journals` работает ТОЛЬКО для `/issues/{id}.json`.**
  В списке `/issues.json` параметр молча игнорируется — ни ошибки, ни
  поля в ответе. Отсюда N+1 в `_fetch_updated_issues`. Проверено на
  живом Redmine 6.x.

- **Вложение — это `details`, а не `notes`.** `property="attachment"`,
  `new_value=<имя файла>`. При удалении файла имя лежит в `old_value` —
  фильтруй по `new_value`, иначе «удалил» приедет как «приложил».

- **`trim_blocks=True` съедает перевод строки после блочного тега.**
  Строка, заканчивающаяся на `{% endif %}`, слипается со следующей.
  Для inline-условий бери условное выражение
  (`{{ a if cond else b }}`) — оно не блочный тег.

- **Тесты конфига читают личный `.env` разработчика.** `Settings` ищет
  его относительно CWD. Добавил переменную в свой `.env` — уронил тест
  дефолтов, хотя код не трогал. Изолируй `monkeypatch.chdir(tmp_path)`.

- **SQLite не хранит смещение времени.** `PollingState.last_check_at`
  объявлен без `DateTime(timezone=True)`, да и в SQLite `DATETIME` —
  обычная строка. Из БД время приезжает **naive**, а `poll_recent_changes`
  сравнивает окно с aware `created_on` из Redmine. Первый цикл прошёл бы
  (курсор пустой, время из `now`), а второй упал бы с `TypeError`.
  Нормализуем в `polling_state._ensure_aware`. Проверено мутацией:
  без него из БД приезжает `tzinfo=None`.

- **Гранулярность таймера в Windows ~15.6мс.** `asyncio.sleep(0.06)`
  просыпается где-то между 0.046 и 0.078. Тест TTL с зазором 10мс
  (`SHORT_TTL=0.05`, сон `0.06`) флакал: сон иногда заканчивался ДО
  истечения TTL. Зазор в тестах со временем — минимум четырёхкратный.

- **SQLAlchemy сам превращает `column == None` в `IS NULL`.** Якорь 4.10
  (NULL != NULL) бьёт по `UniqueConstraint`, но НЕ по `SELECT`'у: там
  перегрузка оператора отрабатывает корректно. Выломанный `.is_(None)`
  тест не заметил. Явную ветку в `is_already_sent` оставили ради
  читаемости, но не думай, что без неё дедуп сломается.

- **`/issues.json` по умолчанию отдаёт ТОЛЬКО открытые задачи.**
  Без `status_id="*"` в фильтрах уведомление «задачу закрыли» не придёт
  никогда: к моменту опроса задача уже закрыта и в выборку не попадает.
  Самое обидное молчание из возможных — и в логах ни следа. См.
  `poller._fetch_updated_issues`.

- **`private_notes` прячет текст заметки, но НЕ изменения атрибутов.**
  Redmine показывает `details` приватной journal-записи всем, скрыт
  только `notes`. Поэтому фильтр приватности вешаем на комментарий,
  а не на запись целиком — иначе молча теряем легитимные события
  «статус изменён».

- **`pytest-httpx` требует, чтобы КАЖДЫЙ мок был запрошен.** Мок, который
  по логике может не сработать (например, `/issue_statuses.json` —
  резолвер идёт в сеть только при смене статуса), помечай
  `is_optional=True`. Иначе тест проходит, но падает на teardown.

- **Async-тест на мгновенном моке не проверяет конкурентность.**
  Тест «пять корутин промахнулись → один HTTP-запрос» был зелёным даже
  с выломанной двойной проверкой в `StatusResolver._refresh`. Причина:
  `httpx_mock.add_response` отвечает без реальной точки переключения,
  первая корутина проходит весь `_refresh`, ни разу не отдав управление
  event loop'у, остальные стартуют уже на заполненном кэше и до lock'а
  не доходят. Запрос один — но по другой причине. Лечится
  `httpx_mock.add_callback(...)` с `await asyncio.sleep(0.02)` внутри
  (+ `is_reusable=True`, иначе второй запрос упадёт раньше assert'а).
  **Мораль:** написал async-тест на гонку — выломай фичу и убедись,
  что он краснеет. Зелёный тест сам по себе не доказательство.
  См. `tests/redmine/test_status_resolver.py`.

- **`check.sh` не видит untracked-файлы.** pre-commit ходит только по
  файлам, известным git'у. Новый файл, не прошедший `git add`, ruff/mypy
  **не проверяют** — `check.sh` зелёный, а `git commit` тут же краснеет
  (поймали на E501 в `status_resolver.py`). Перед `check.sh` на новых
  файлах — сначала `git add`.

- **Часы в тестах не мокаются глобально.** `time.monotonic` — общий
  с event loop'ом, который меряет им свои таймеры. Заморозишь через
  monkeypatch — `asyncio.sleep` не проснётся никогда. В тестах TTL берём
  микроскопический (0.05с) и спим по-настоящему.

- **`AsyncIOScheduler.shutdown()` — асинхронный по факту.** Метод
  декорирован `@run_in_event_loop`, реальный переход в
  `STATE_STOPPED` происходит не при возврате, а в следующей итерации
  event loop'а (через `call_soon_threadsafe`). В проде — 0 проблем
  (loop гасится сразу). В тестах приходится дожимать через
  `await asyncio.sleep(0)` в цикле «до N тиков», иначе проверка
  `scheduler.running is False` даёт ложный True. См. пример в
  `tests/web/test_scheduler_lifespan.py`.

- **`apscheduler` не типизирован.** В `pyproject.toml` секция
  `[[tool.mypy.overrides]] module=["apscheduler.*"]` с
  `ignore_missing_imports=true`. Убрать, если появятся стабы.

- **mypy в pre-commit — изолированная venv.** Каждая новая рантайм-
  или тест-зависимость **обязана** быть в `additional_dependencies`
  mypy-хука в `.pre-commit-config.yaml`. Иначе `import-untyped` /
  `import-not-found`.

- **pre-commit стэшит несостейдженные правки.** Если
  `pre-commit run --all-files` зелёный, а `git commit` красный —
  почти всегда правки конфигов не застейджены. Лечится
  `git add <файл-конфига>` и повторным коммитом.

- **CC требует VPN для API.** `api.anthropic.com` отдаёт 403 с
  российского IP (Cloudflare-edge геоблокирует). Запуск CC — только
  через алиас `claude-vpn`, который выставляет HTTPS_PROXY на локальный
  порт HAPP + NO_PROXY для внутренней сети (192.168.100.0/24).
  Наш прод-код не пострадает: во всех `httpx.AsyncClient` стоит
  `trust_env=False` (грабля из этапа 2, см. якорь 4.2 в CLAUDE.md).

## @упоминания (сделано сверх плана, 15.07.2026)

Работает: упоминание исполнителя в 🆕 новой задаче и ⏰ напоминании
о дедлайне. Проверено на живом MAX — пинг доходит.

**Формат найден экспериментом, а не по аналогии.** Итог разведки API:

- **`[Имя](max://user/<id>)` — единственное, что работает для людей.**
- `@username` подсвечивается только у **ботов**: у живых людей поля
  `username` в MAX нет вовсе. Телеграмная привычка тут бесполезна.
- `markup` с `user_mention` (по аналогии с VK/OK) MAX **молча
  игнорирует** — вернул 200 и обычный текст.
- `GET /chats` и `GET /chats/{id}/members` работают и дают `user_id`.
- `GET /messages` и `get_updates` истории чата боту **не отдают**
  (0 сообщений при `messages_count: 18`).

**Ключевое ограничение:** бот видит только участников чатов, где сам
состоит. Значит упомянуть можно лишь того, кто в чате. Это логика
мессенджера, не наша.

**Как устроено:**

- таблица `user_mapping` (миграция `b7eac8b24689`), многие-ко-многим
  как у routing: Петя → Петя+тимлид, Петя+Вася → Вася, произвольные
  пары;
- `max_name` — снимок имени из MAX. Ссылка работает по `user_id`,
  ходить за подписью в API на каждое уведомление незачем;
- **резолвит диспетчер**, не детектор: кого пинговать — свойство
  доставки в мессенджер, а не факт из Redmine. Детектор про БД
  по-прежнему не знает;
- имя в упоминании — **MAX-юзера**, не Redmine: при сопоставлении
  «Петя → Вася» в чате виден и дёрнут Вася;
- `format_mention` со **своим** экранированием: фильтр `| md` глушит
  `[` и `]` и развалил бы ссылку;
- не сопоставлен — уведомление уходит без пинга, молча (иначе каждый
  цикл сыпал бы warning'ами про одних и тех же людей).

CLI: `scripts/user_mapping_cli.py` (+ `max-members` и `redmine-users`
как источники id). Памятка — `cheatsheets/mentions.md`.

**Не делаем пока:** упоминания в сменах статуса и комментариях —
чтобы не частить пингами (решено 15.07.2026).
