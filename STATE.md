## Что закрыто в 7c–7d (итог)

**`status_resolver.py`** — TTL-кэш «id статуса → имя». Обновляется
целиком, свежесть на `time.monotonic()`, конкурентные промахи
схлопываются в один HTTP-запрос через `asyncio.Lock` + двойную проверку
внутри лока.

**`poller.py`** — `PollCursor` + `poll_recent_changes`. Чистая функция:
курсор на вход, `(события, новый курсор)` на выход. Ни БД, ни отправки.

Принятые решения (чтобы не переобсуждать):

- **`monotonic`, не `datetime.now()`** — настенные часы прыгают (NTP,
  перевод времени) и ломают арифметику TTL.
- **Ошибка Redmine при протухшем кэше → исключение наружу**, stale-значение
  НЕ отдаём. Решение «ловить или падать» принимает поллер (7f).
- **Промах по свежему кэшу → сразу `None`, без рефреша.** Иначе событие
  с удалённым статусом било бы по API на каждом проходе.
- **Окно `updated_on` выбирает задачи, новизну решают id.** Окно нужно,
  потому что эндпоинта `/journals.json` в Redmine нет. Но `issue.id` и
  `journal.id` глобально монотонны — детекция по ним не зависит от часов.
  Окно поэтому может быть щедрым, лишнее отрежут id.
- **Холодный старт = `last_check_at is None`**, а не пустые `last_seen_*`.
  Первый цикл ставит baseline и не шлёт ничего. Определяй мы холодный
  старт по id, тихий Redmine (пустое окно → максимумов нет) залипал бы
  в «первом запуске» вечно.
- **Удалённый статус → событие пропускается с `log.warning`**, а не
  заглушка «#8» и не падение цикла.

## Задача подэтапа 7e

Идемпотентность + диспетчер отправки. Собираем воедино то, что уже
написано по кускам: событие → проверка «не слали ли» → routing
(проект → чаты) → рендер → MAX → отметка «отправлено».

### Что нужно сделать

1. **CRUD идемпотентности** над `SentNotification`:
   `is_already_sent(session, event) -> bool` и `mark_sent(session, event,
   chat_id)`. Модуль — по аналогии с `routing.py` (проверить на входе,
   где лежит routing-CRUD, и положить рядом).
2. **Якорь 4.10 — на первом месте.** `UNIQUE(event_type, issue_id,
   journal_id)` НЕ ловит дубли при `journal_id IS NULL` (NULL != NULL
   по SQL-стандарту). Для `new_issue` (и `due_date` на 7g) —
   **отдельный SELECT** перед вставкой, полагаться на констрейнт нельзя.
3. **Диспетчер** `dispatch(event, session, renderer, max_client) -> None`:
   routing по `event.issue.project.id` → для каждого чата рендер +
   `send_message` + `mark_sent`.

### Тонкости для обсуждения на входе

- **Порядок «отметить/отправить».** Отметить ДО отправки — риск потерять
  уведомление (упали между). Отметить ПОСЛЕ — риск дубля при рестарте.
  Склоняюсь к «после»: дубль в чате раздражает, потерянное уведомление
  о проде — вредит. Но обсудить.
- **Гранулярность отметки.** `SentNotification` не хранит `chat_id`,
  значит отметка «событие отправлено» — на событие целиком, а не на
  пару (событие, чат). Если чатов у проекта два и второй упал —
  повторять оба или ни одного? Возможно, схему придётся расширить.
  **Разобраться до кода.**
- **Пустой routing.** У проекта нет чатов → событие отправлять некуда.
  Молча пропускать или `log.warning`? Склоняюсь к warning: скорее
  всего, админ забыл прописать routing для нового проекта.
- **Падение отправки в один чат** не должно ронять остальные адресаты
  и весь цикл.

## Активные грабли (последние 2 этапа)

Свежие наблюдения. Когда этап закроется целиком и «повторяющаяся»
грабля себя не проявит ещё пару этапов — переносим её в `CLAUDE.md`
или просто удаляем.

- **`/issues.json` по умолчанию отдаёт ТОЛЬКО открытые задачи.**
  Без `status_id="*"` в фильтрах уведомление «задачу закрыли» не придёт
  никогда: к моменту опроса задача уже закрыта и в выборку не попадает.
  Самое обидное молчание из возможных — и в логах ни следа. См.
  `poller._fetch_updated_issues`.

- **`private_notes` прячет текст заметки, но НЕ изменения атрибутов.**
  Redmine показывает `details` приватной journal-записи всем, скрыт
  только `notes`. Поэтому фильтр приватности вешаем на комментарий,
  а не на запись целиком — иначе молча теряем легитимные события
  «статус изменён».

- **`pytest-httpx` требует, чтобы КАЖДЫЙ мок был запрошен.** Мок, который
  по логике может не сработать (например, `/issue_statuses.json` —
  резолвер идёт в сеть только при смене статуса), помечай
  `is_optional=True`. Иначе тест проходит, но падает на teardown.

- **Async-тест на мгновенном моке не проверяет конкурентность.**
  Тест «пять корутин промахнулись → один HTTP-запрос» был зелёным даже
  с выломанной двойной проверкой в `StatusResolver._refresh`. Причина:
  `httpx_mock.add_response` отвечает без реальной точки переключения,
  первая корутина проходит весь `_refresh`, ни разу не отдав управление
  event loop'у, остальные стартуют уже на заполненном кэше и до lock'а
  не доходят. Запрос один — но по другой причине. Лечится
  `httpx_mock.add_callback(...)` с `await asyncio.sleep(0.02)` внутри
  (+ `is_reusable=True`, иначе второй запрос упадёт раньше assert'а).
  **Мораль:** написал async-тест на гонку — выломай фичу и убедись,
  что он краснеет. Зелёный тест сам по себе не доказательство.
  См. `tests/redmine/test_status_resolver.py`.

- **`check.sh` не видит untracked-файлы.** pre-commit ходит только по
  файлам, известным git'у. Новый файл, не прошедший `git add`, ruff/mypy
  **не проверяют** — `check.sh` зелёный, а `git commit` тут же краснеет
  (поймали на E501 в `status_resolver.py`). Перед `check.sh` на новых
  файлах — сначала `git add`.

- **Часы в тестах не мокаются глобально.** `time.monotonic` — общий
  с event loop'ом, который меряет им свои таймеры. Заморозишь через
  monkeypatch — `asyncio.sleep` не проснётся никогда. В тестах TTL берём
  микроскопический (0.05с) и спим по-настоящему.

- **`AsyncIOScheduler.shutdown()` — асинхронный по факту.** Метод
  декорирован `@run_in_event_loop`, реальный переход в
  `STATE_STOPPED` происходит не при возврате, а в следующей итерации
  event loop'а (через `call_soon_threadsafe`). В проде — 0 проблем
  (loop гасится сразу). В тестах приходится дожимать через
  `await asyncio.sleep(0)` в цикле «до N тиков», иначе проверка
  `scheduler.running is False` даёт ложный True. См. пример в
  `tests/web/test_scheduler_lifespan.py`.

- **`apscheduler` не типизирован.** В `pyproject.toml` секция
  `[[tool.mypy.overrides]] module=["apscheduler.*"]` с
  `ignore_missing_imports=true`. Убрать, если появятся стабы.

- **mypy в pre-commit — изолированная venv.** Каждая новая рантайм-
  или тест-зависимость **обязана** быть в `additional_dependencies`
  mypy-хука в `.pre-commit-config.yaml`. Иначе `import-untyped` /
  `import-not-found`.

- **pre-commit стэшит несостейдженные правки.** Если
  `pre-commit run --all-files` зелёный, а `git commit` красный —
  почти всегда правки конфигов не застейджены. Лечится
  `git add <файл-конфига>` и повторным коммитом.

- **CC требует VPN для API.** `api.anthropic.com` отдаёт 403 с
  российского IP (Cloudflare-edge геоблокирует). Запуск CC — только
  через алиас `claude-vpn`, который выставляет HTTPS_PROXY на локальный
  порт HAPP + NO_PROXY для внутренней сети (192.168.100.0/24).
  Наш прод-код не пострадает: во всех `httpx.AsyncClient` стоит
  `trust_env=False` (грабля из этапа 2, см. якорь 4.2 в CLAUDE.md).
